# 用户体验框架（页面 · 功能 · MVP 与实现对照）

本文档描述**信息架构（IA）**、**各页能力**以及**产品 MVP 必做**与**当前仓库代码**的对照，供产品、设计、前端、测试与 AI 协作时对齐使用。

若需**单文件**交给外部 AI 做前端（含接口与体验约束），请使用 **`docs/产品功能与体验规格-AI前端用.md`**。

---

## 1. 文档定位：正常开发里它长什么样、怎么用

在常见团队里，这类材料会以不同**粒度**出现，本文件把它们**收拢成一份 Markdown**，避免散落在聊天记录里：

| 呈现形式 | 典型用途 |
|----------|----------|
| **PRD / 产品描述附录** | 与 `docs/产品描述.md` 配套：PRD 写规则与边界，本文写「有哪些屏、每屏干什么」 |
| **设计交付索引** | Figma/蓝湖按页面分文件，本文列出页面清单与状态（空态/加载/错误/已配对）便于走查 |
| **前端路由/目录清单** | 与 `backend/public/*.html` 或未来 SPA 路由表对齐，改 IA 时先改本文再改代码 |
| **测试用例大纲** | 按「页面 × 功能 × 必做/已有」拆用例，覆盖回归范围 |
| **迭代排期表输入** | 「必做未实现」行直接进 Backlog；「已有」行写回归即可 |

**优先级说明**：业务规则与数值以 **`docs/产品描述.md`** 为准；若本文与产品描述冲突，**以产品描述为准**并应回写修正本文。工程是否落地以 **`ARCHITECTURE.md` 第 11 节** 为准。

## 2. 图例（下文表格统一使用）

| 标记 | 含义 |
|------|------|
| **MVP 必做** | 产品已定、上线前应交付的用户可感知能力（依据 `docs/产品描述.md` 核心旅程与规则） |
| **已有** | 当前仓库中已有**可跑通的主路径**（页面或接口 + 前端联调） |
| **部分** | 有页面或接口骨架，但与 MVP 规则未完全对齐或缺关键后端 |
| **未做** | 尚未在主线代码中实现 |

## 3. 页面地图（用户动线）

```text
注册 / 登录
    ↓
首页（发现主题）───→ 预约场次（选时间 + 水平 + 确认）
    ↓                        ↓
我的预约（列表 / 状态 / 搭档 / 进房）──→ 练习房间（RTC + 任务 + 结束）
```

**非用户主路径**：`agora-test.html` 为声网集成调试页，不计入产品用户体验主线。

## 4. 按页面：功能清单与 MVP / 实现对照

### 4.1 注册 / 登录

| 功能点 | MVP 必做 | 当前代码 |
|--------|----------|----------|
| 用户可注册并建立账户 | 是 | **已有**：`register.html`（昵称、邮箱、密码） |
| 用户可登录并保持会话 | 是 | **已有**：`login.html` + Cookie JWT；支持 `?next=` 回跳 |
| 登录方式符合产品主路径 | 是（产品：**手机号 + 短信验证码**，邮箱为 MVP 备用） | **部分**：仅 **邮箱 + 密码**，无短信验证码 |
| 首次登录引导昵称、初始信用分 10 | 是 | **部分**：注册时昵称已有；信用分字段存在，**完整展示/禁用预约等规则闭环见房间与结算** |

### 4.2 首页（发现主题）

| 功能点 | MVP 必做 | 当前代码 |
|--------|----------|----------|
| 浏览可练习主题并进入预约 | 是 | **已有**：`index.html` 经 `GET /api/themes` 动态渲染三卡 → `booking.html?theme_id=…` |
| 周主题轮换（每周日 19:00 起开放下一周三主题） | 是 | **已有**：`weekThemeCycle.js` + 库表轮换；非运营后台 GUI |
| 展示当前用户信用分（若已登录） | 是（个人感知） | **已有**：`home-session.js` 拉 profile |
| 底栏主导航（首页 / 我的预约） | 建议 | **已有** |

### 4.3 预约场次

| 功能点 | MVP 必做 | 当前代码 |
|--------|----------|----------|
| 选择主题下的时间段 | 是 | **已有**：`booking.html?theme_id=` + `booking-flow.js` 拉 `GET /api/timeslots?theme_id=` |
| 选择口语水平（初/中/高） | 是 | **已有** |
| 展示该场已约人数及水平分布（不暴露是谁） | 是 | **部分**：以接口/数据为准，若后端未返回统计则需补 |
| 确认预约并写库 | 是 | **已有**：`POST /api/bookings` |
| 开场前 **60 分钟**停止新预约与**首次配对**（取消重配除外） | 是 | **未做**：预约事务未按时间拦截；无配对调度 |
| 同用户不可重复约同一场 | 是 | **已有**：库唯一约束 + 接口 |
| 展示匹配规则摘要 / 链到帮助 | 建议 | **部分**：预约页有句文案；**独立帮助页未做** |

### 4.4 我的预约

| 功能点 | MVP 必做 | 当前代码 |
|--------|----------|----------|
| 仅展示当前用户自己的预约 | 是 | **已有**：`GET /api/bookings/mine` + `appointments.html` |
| 未配对 / 已配对 / 对方取消重配 等状态文案 | 是 | **部分**：列表与轮询有，**须与 `产品描述` 5.2～5.3 对齐迭代** |
| 配对后立即展示搭档昵称与水平（不展示手机/邮箱）；取消后重配状态 | 是 | **部分**：`pairs` 在开场窗口内由 `runAutoPairingScan` 写入；通知与重配文案仍待对齐 |
| 进入房间的入口（链接/按钮） | 是 | **已有**：`room.html?timeslot_id=` + `rtc-token-booking`（等待大厅；配对后切 1v1） |
| 取消预约（若产品允许时间窗） | 视产品细则 | **已有**：`DELETE /api/cancel-booking/:id`（`cancelBooking.js`）；`appointments-live.js` 二次确认；**开场时间到达后**不可取消 |
| 开发调试：一键写入 `pairs` 测进房 | 否（研发用） | **已有**：非生产或 `ENABLE_DEV_PAIRING=1` 时 `POST /api/dev/pair-timeslot`（见 `ARCHITECTURE` 5.2） |

### 4.5 练习房间

| 功能点 | MVP 必做 | 当前代码 |
|--------|----------|----------|
| 展示任务卡、角色、计时 | 是 | **已有**：`room.html` 结构 + 脚本 |
| 实时语音（Agora），频道名服务端下发 | 是 | **部分**：`rtc-token-booking` + 轮询切频道**已有**；**生产级「用户属于该 pair」校验待补** |
| 展示对方昵称等非敏感信息 | 是 | **部分**：演示文案/Query 与正式 `my-pair` 分流需统一 |
| 可选视频位 / iOS 能力提示 | 建议 | **已有**：顶栏提示 + 视频占位 |
| 子任务列表、完成确认、互换角色显示（扩展） | 可选迭代 | **部分**：`room-tasks.js`、WebSocket 演示；产品细则以交互稿与接口为准 |
| 满 10 分钟、双方确认结束、信用分结算 | 是 | **未做**：无完整结算 API 与 `credit_logs` 业务闭环 |

### 4.6 系统能力（不一定独占一页）

| 功能点 | MVP 必做 | 当前代码 |
|--------|----------|----------|
| 尽早配对、取消重配、停配扫描、开场落单互配、写 `pairs`、兜底 | 是（`产品描述` 5.3、7、9 节） | **未做**：无事件/定时配对任务 |
| 配对与状态通知（邮件优先 MVP） | 是 | **未做** |
| 无法全员配对时的场次取消与补偿 | 是 | **未做** |
| 数据隔离：鉴权后按 `user_id` 查询 | 是 | **已有**：中间件 + bookings 查询模式；**新增接口须延续** |

## 6. 维护约定

- 产品规则变更：先改 **`docs/产品描述.md`**，再更新**本文件第 4～5 节**对照关系，并在 **`docs/产品与需求变更规则.md`** 第 4 节追加一行。
- 仅实现进度变化（例如「未做」→「已有」）：更新本文件表格即可，**建议**仍在变更记录里记一笔便于追溯。
