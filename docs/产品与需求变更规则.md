# 产品与需求变更规则

本文档约定：**凡是本产品（英语口语真人匹配练习平台）的需求、规则或业务结论发生变化，必须先改文档，再（或同时）改代码。**  
避免「口头改了需求、仓库里的说明仍是旧版」导致实现与产品不一致。

---

## 1. 文档分工

| 文档 | 路径 | 用途 |
|------|------|------|
| **产品描述** | `docs/产品描述.md` | 功能范围、用户旅程、配对与信用分规则、兜底策略、通知与主题等**正式产品说明** |
| **对抗式任务与界面** | `docs/对抗式任务玩法与界面.md` | 子任务确认制、能量与轮次、收银员/客户示例、移动端不打断对话的 UI 原则 |
| **架构与开发规范** | `ARCHITECTURE.md` | 技术栈、目录、数据库、API、工程侧硬约束（与产品描述对齐） |
| **本变更规则** | `docs/产品与需求变更规则.md` | 何时改文档、改哪些文档、**变更记录**（摘要留痕） |

新增独立说明（例如 API 约定、数据库设计）时，在下方「变更记录」里记一笔，并在本节表格中补充一行。

---

## 2. 什么情况必须更新文档

在以下任一情况发生时，**必须**更新 `docs/产品描述.md`（或新建专项文档并在本文件记录路径），并在**第 4 节变更记录**追加一条：

- 功能增删或流程变化（登录、预约、配对、房间、信用分、通知等）
- 业务规则变化（例如配对约束、加减分、阈值、兜底策略、文案策略）
- 目标用户或范围变化（如地域、MVP 边界）
- 与实现强相关的**产品决策**（例如虚拟语伴是否做、优先预约权如何定义）

**仅修 UI 样式、重构代码而不改行为**时，不必改产品描述，除非交互改变了用户可感知流程。

---

## 3. 推荐工作流

1. **先改** `docs/产品描述.md`：把新结论写进对应章节，删除或标注废止旧表述。  
2. **再改** 本文件第 4 节：用一行或一小段说明**改了什么、日期**。  
3. **最后** 再改代码 / 前端页面，使行为与文档一致。  
4. 若在 Cursor 中与 AI 协作：说明「已按 `docs/产品描述.md` 第 × 节为准」，避免模型沿用旧对话记忆。

若某次变更只涉及「变更流程本身」（例如要多维护一份 API 文档），请在本文件 **第 1 节表格** 或 **第 2 节** 中同步修改说明。

## 5. 与 Cursor 规则的关系

项目内 `.cursor/rules/` 中的相关规则会提示：在对话中修改产品需求时，应**同步更新** `docs/产品描述.md` 与本文件的**变更记录**。  
若你希望某条规则始终生效，请保留该 `.mdc` 文件；若仅人工维护文档，也可只依赖本 Markdown 流程。
