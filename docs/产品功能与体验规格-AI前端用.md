# 英语口语真人匹配练习平台 — 产品功能与体验规格（供前端 / AI 实现用）

> **用途**：将本产品「功能 + 体验 + 页面 + 接口 + 前端约束」合并为**单份**说明，可直接作为 Prompt 附件交给另一 AI 做界面或重构前端。  
> **权威来源**：业务规则以 `docs/产品描述.md` 为准；若本文与之一致性有误，以 `产品描述.md` 为准并回改本文。  
> **后端路径**：当前仓库为 **Express + 静态 HTML/JS**（`backend/public/`）；重做前端时接口路径须与下表「已实现」一致，除非同步改后端。

---

## 1. 产品一句话

面向**中国大陆用户**的**手机浏览器优先**的 Web 应用：用户**预约主题 + 场次时间 + 自选口语水平**，在约定时间进入**角色扮演式英语对话房间**，与**真人**语伴练习口语；系统在**同主题+场次+水平可配时尽早配对**并**立即**告知搭档；**开场前 1 小时**停新约与首次配对；**开场**可对队友未上线者**重配**；使用 **RTC（声网 Agora）** 实时语音（可选视频）；**信用分**约束履约。

## 3. 登录与账户（产品 vs 当前后端）

| 产品（目标） | 当前已实现 API / 行为 |
|--------------|------------------------|
| **手机号 + 短信验证码**为主；邮箱验证码可作 MVP 备用 | **邮箱 + 密码**；`POST /api/auth/register`、`POST /api/auth/login`；会话为 **HttpOnly Cookie JWT** |

## 4. 水平等级（预约与配对）

| 产品文案 | 前端预约可用值（示例） | 提交给 API 的 `level`（当前仓库） |
|----------|------------------------|-----------------------------------|
| 初级 | `beginner` | `beginner` |
| 中级 | `mid` | 映射为 **`intermediate`** |
| 高级 | `adv` | 映射为 **`advanced`** |

水平用于：**场次内人数/水平分布展示**、**自动配对**（两人等级差 **≤ 1**，不允许初–高）。**不向其他用户展示手机号、邮箱。**

## 5. 核心用户旅程（必须能在 UI 上走通）

### 5.1 预约

1. 选择**主题**与**时间段**。  
2. 选择**口语水平**（初/中/高）。  
3. 展示该场 **已约人数及水平分布**（例：`已约：初 3，中 2，高 1`），**不展示具体是谁**。  
4. 用户确认后生成预约。

**强制规则（产品）**：

- **周主题轮换**：**每周日 19:00（北京时间）起**开放**下一自然周**的 **3 个**练习主题；每个主题仅在该周有效，至该周**周日最后一场**结束后由服务端轮换为下周三主题。**周日 19:00～21:00** 与「本周」周期尾可能重叠，此间接口与首页以**下一自然周**为准。首页 `GET /api/themes`，预约页 `booking.html?theme_id=`。场次仍为每主题**该周周六、日 20:00**（`GET /api/timeslots?theme_id=` 过滤展示）。  
- 距场次开始 **不足 60 分钟**：**停止接受新预约**，并**停止对尚未首次配对的用户做新的首次配对**；列表须显示不可约原因；**已确认预约不因此自动取消**（除非产品另定）。  
- **同一用户不可重复预约同一时间段**；可有多个不同场次的未开始预约。  
- **预约页主文案（可缩为一句）**：  
  > 同一**主题**、**场次时间**下按**水平**（差距不超过一级）**尽早配对**，一旦成功会**立刻**告知**搭档昵称与水平**。**开场前 1 小时**停止新预约与首次配对；该时段内用户**应已知当前搭档**。若搭档**取消**，另一方会收到「正在为你重新安排搭档」类提示。

详细配对规则可放「帮助中心」（可后续单独页或折叠）。

### 5.2 预约后至开场前（我的预约 / 详情）

- 展示该场 **总人数 + 水平分布**（同预约页逻辑）。  
- **已配对**：立即展示搭档 **昵称 + 口语水平**（与通知一致）；**不展示**对方手机/邮箱。  
- **未配对**：展示匹配中状态，不展示具体是谁。  
- **对方取消导致你落单**：展示「原搭档已取消，正在为你匹配另一位搭档」类状态（与通知一致）。

**状态文案（产品示例）**：

| 条件 | 提示文案（示例） |
|------|------------------|
| 未配对 | 正在为你寻找水平合适的搭档，请留意通知与个人中心 |
| 已配对 | 展示搭档昵称与水平；如「已配对，开场前请完成准备」 |
| 对方取消 | 你原先的搭档已取消预约，正在为你重新安排另一位搭档 |

距开场不足 60 分钟：**新用户**不可再约该场；**已约用户**仍可见预约与配对状态；**取消触发的重配**仍应继续尝试。

### 5.3 配对、停配、取消与开场互配（系统行为，影响 UI）

- **尽早配对**：同 **主题** + **场次** 下，等级满足硬约束即可写入 `pairs`，并**立即**通知（邮件/短信/Push 分阶段）与更新个人中心。  
- **一方取消**：另一方收到「正在重配」类通知与页面状态。  
- **停配时刻**（开场前满 60 分钟）：做全员是否可合法成对检查；**若无法全员成对**则场次取消与补偿（见 `docs/产品描述.md` 第 7 节）；前端须能展示「场次已取消」等。  
- **开场到点**：对「已上线但队友未上线」的用户做**落单池互配**，并通知：「您的队友未上线，已为您分配新的练习搭档」（措辞可本地化）。

### 5.4 房间与练习

- 展示：**对方昵称**、**任务卡**（双方角色与信息差，与主题一致）、**计时器**。  
- **语音**为主，RTC 使用 **Agora**；**频道名、Token 必须由服务端下发**，前端**禁止自拟频道**以免串线。  
- **可选**：简单「互换角色显示」、子任务列表等可在 MVP 后迭代；**不再使用已移除的独立对抗式文档**作为需求来源。

### 5.5 结束与信用分（界面须预留）

- **满 10 分钟**后，方可按产品流程结束；**双方确认完成**后计分。  
- 爽约、单方早退、守约补偿等见下表。  
- **信用分 ≤ 3**：**禁止预约**，需联系管理员恢复（前端可禁用预约按钮并提示）。

| 情形 | 分数 |
|------|------|
| 正常完成（满 10 分钟且双方确认） | 双方各 **+1** |
| 爽约 | **-2** |
| 中途提前退出（未满 10 分钟且无双方同意） | 责任方 **-1**（守约方不按本条扣） |
| 对方爽约或对方违规早退 | **守约方 +1** |
| 因平台无法全员配对而取消场次 | **不扣用户分**；补偿每位 **+2**（防重复发放由后端保证） |

## 6. 主题与任务卡（产品）

- **主题**：MVP 为 **5～10 个固定主题**，由运营/后台维护；前端拉列表展示（当前仓库有种子数据 + 首页写死入口时可过渡）。  
- **任务卡**：可由 LLM 生成；**同一主题 MVP 共用一套任务卡**；房间内展示**双方角色与信息差**。  
- **迭代**：可按水平生成不同难度任务卡（后期）。

## 7. 信息架构（页面清单）

以下为**用户主路径**；调试页不计入产品。

| 顺序 | 页面 / 路由（概念） | 核心能力 |
|------|---------------------|----------|
| 1 | **注册** | 昵称、账号凭证、同意条款（若有）、注册成功后登录态 |
| 2 | **登录** | 登录、错误提示、登录后跳回 `next`（若合法相对路径） |
| 3 | **首页 · 发现主题** | 主题列表/卡片、展示信用分、进入预约、底栏「我的预约」 |
| 4 | **预约场次** | 选日期/场次、选水平、看分布与规则摘要、确认预约、不可约态 |
| 5 | **我的预约** | 仅本人列表、状态与文案、配对后搭档信息、进房入口、下拉刷新或轮询 |
| 6 | **练习房间** | 计时、任务卡、对方昵称、RTC 控制（静音/挂断等按设计）、iOS 提示 |
| 7 | **结束 / 反馈**（可与房间合并为流程） | 满 10 分钟提示、双方确认、展示结果与信用分变化（依赖后端） |

**可选**：帮助中心 / 关于我们（长规则折叠展示）。

## 8. 数据隔离与隐私（前端配合）

- 列表与详情只展示**当前登录用户**的数据；**不要**把「当前用户 id」放在请求体里冒充他人（后端应以 Cookie 会话为准）。  
- UI 上**永远不要**展示其他用户的手机号、邮箱。

## 9. 后端接口（当前仓库已实现，前端应对接）

**Base URL**：与部署同源或配置的 API 根路径。  
**认证**：需要登录的接口带 **`credentials: 'include'`**（Cookie）。

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/auth/register` | Body: `email`, `password`, `nickname` |
| POST | `/api/auth/login` | Body: `email`, `password` |
| POST | `/api/auth/logout` | 清除会话 |
| GET | `/api/auth/me` | 当前用户；未登录返回 401 |
| GET | `/api/themes` | 当前开放周期内三个主题（含 `id`、封面、场景、角色 JSON 等） |
| GET | `/api/themes/by-id` | Query: **`id`**=`theme_id`，预约页展示用 |
| GET | `/api/timeslots` | Query: **`theme_id`（必填）** |
| POST | `/api/bookings` | Body: `timeslot_id`, `level`（`beginner` \| `intermediate` \| `advanced`） |
| GET | `/api/bookings/mine` | 我的预约列表；含 `partnerNickname`、`channelName`、`pairStatus` 等（配对存在时） |
| DELETE | `/api/cancel-booking/:id` | 取消本人预约（开场前）；服务端删除 `bookings` 行，故**可再次预约同一场次** |
| POST | `/api/agora/rtc-token` | Body: `channelName`, `uid`；返回声网入会信息 |

**统一响应形态**（常见）：`{ code, message, data }`；`code === 0` 为成功。

**尚未实现或仅部分实现（前端需容错 / 占位）**：

- 产品要求的 **`GET /api/my-pair`**、**`POST /api/end-conversation`**、**取消预约** 等与 `ARCHITECTURE.md` 第 5 节命名可能不一致，**以仓库 `backend/routes` 为准**。  
- **开场前 60 分钟截止预约**、**配对 Cron**、**邮件通知** 等可能未完备；UI 仍应按产品展示不可约原因、加载中与错误态。

## 10. 前端工程与视觉约束（与架构一致）

| 项 | 要求 |
|----|------|
| 视口 | `<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">` |
| 触控 | 主要点击区域 **≥ 44×44px** |
| 主视口 | 手机优先，主内容区宽度约 **&lt; 600px** 设计基线 |
| 色彩 | 应用内统一蓝色系（如 `#1A5CFF` / `#7096FF` / `#4A90E2` 择一并全站统一） |
| 圆角 | 卡片约 **12px** |
| 反馈 | 加载用 spinner/骨架；错误用 **toast**；避免主流程依赖原生 `alert` |
| HTTP | 使用 `fetch`；注意 Cookie 跨域时需后端 CORS `credentials` |
| RTC | 使用 **Agora Web SDK 4.x**；先取 Token 再 `join` |

## 11. 给「接手的 AI」的实现顺序建议

1. 布局壳 + 底栏导航 + 全局字体/色板。  
2. 登录 / 注册 / `me` 拉信用分。  
3. 首页主题列表（可先写死或接 DB 扩展）。  
4. 预约页：`GET /api/themes/by-id` 渲染头图与场景 → `timeslots?theme_id=` 选档 → `POST /bookings`。  
5. 我的预约：`GET /api/bookings/mine` + 状态文案 + 进房（无 `channelName` 时置灰）+ `DELETE /api/cancel-booking/:id` 取消（二次确认）。  
6. 房间页：展示 + `rtc-token` + 加入频道 + 麦克风（可选摄像头）+ iOS 横幅。  
7. 结束双确认、信用分展示（待后端接口落地后对接）。
