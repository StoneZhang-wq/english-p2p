# 对抗式角色任务玩法与移动端界面方案

本文档描述一种**零语音识别成本**的房间内玩法：**执行方点击完成任务 → 对方限时确认/否认**，配合能量、轮次与角色互换。可与通用「任务卡 + 口语对话」并存，优先适用于**对抗类主题**（如银行取钱：收银员 vs 客户）。

**关联文档**：预约、配对、信用分总规则见 `docs/产品描述.md`；工程实现见 `ARCHITECTURE.md`。

---

## 1. 核心判断逻辑：手动触发 + 对方确认

| 环节 | 规则 |
|------|------|
| 触发 | 每个**子任务**由**执行方**主动点击「完成任务」类按钮。 |
| 对方动作 | 对方在**一段时间内**（建议 **10 秒**，可配置）选择 **「确认」** 或 **「否认」**。 |
| 超时 | 对方**确认**或**超时未操作** → 视为**任务成功**，执行方获得**积分/能量**（数值可配置，示例见第 2 节）。 |
| 否认 | 对方**否认** → 任务**失败**；可记录**争议**；MVP 可**信任用户**，仲裁、申诉后置。 |

**优点简述**

- 无语音识别成本，实现简单。  
- 用户有掌控感，同时需要倾听、判断对方是否真完成。  
- 与现有**信用分**机制结合，可约束恶意否认（见第 6 节）。

## 2. 对抗任务示例：收银员（A）vs 客户（B）

**第一轮：角色固定**

- **角色 A — 收银员**：大目标为**阻止客户取钱**。  
- **角色 B — 客户**：大目标为**坚持取钱**。  

**能量规则（示例）**：每完成一个子任务 **+10 能量**；最终胜负额外奖励见第 2.3 节。

### 2.1 角色 A（收银员）子任务

| 序号 | 子任务 | 触发方式 | 对方确认逻辑 |
|------|--------|----------|--------------|
| ① | 主动询问取钱用途 | A 点击「询问用途」 | B 确认「他问了」（通常无动机否认） |
| ② | 提出一个替代方案（如理财） | A 点击「提出替代方案」 | B 确认「他提了」 |
| ③ | 让客户犹豫一次 | A 点击「客户犹豫了」 | B 须点击「是的，我犹豫了」才算成功；B 可不承认，但影响**合作分/信任**（产品可配） |
| 最终 | （见下） | 不适用单独按钮链 | 由 B 的最终决策结束本轮（见 2.3） |

### 2.2 角色 B（客户）子任务

| 序号 | 子任务 | 触发方式 | 对方确认逻辑 |
|------|--------|----------|--------------|
| ① | 明确说出取钱金额 | B 点击「说出金额」 | A 确认「他说了金额」 |
| ② | 反驳一次替代方案 | B 点击「反驳」 | A 确认「他反驳了」 |
| ③ | 询问另一家银行的利率 | B 点击「询问利率」 | A 确认「他问了」 |
| 最终 | 取/不取决策 | B 点击最终按钮（见 2.3） | **无需对方确认**（决策权在 B） |

### 2.3 最终任务（第一轮结束）

- B 的页面提供两个主按钮：**【我决定取钱】**、**【我决定不取】**。  
- B 点击其一后，**第一轮对话阶段结束**，进入**结算**。  
- **胜负（示例）**  
  - B 选**取钱** → B 赢得对抗，A 失败（B 可获额外奖励如 **+50 能量**，A 无此奖励或较少）。  
  - B 选**不取** → A 成功阻止，A 获额外奖励（如 **+50 能量**）。  
- **说明**：最终按钮不要求 A 确认，避免剥夺客户决策权；与「子任务需确认」刻意区分。

---

## 3. 第二轮：角色互换

1. 第一轮结算完成后，**系统自动交换角色**：原 A → 客户，原 B → 收银员。  
2. 使用**同一套任务结构**（或仅调整子任务顺序/文案微调），进行**第二轮对话**。  
3. 建议**与第一轮相同的单轮时长限制**（如 10 分钟），与产品总规则对齐时可单独约定。  
4. **第二轮结束**后：比较**两轮总分**（子任务能量 + 最终胜负奖励）。  
5. **总分高者**为整局胜者，可获得**双倍能量**或**特殊勋章**等（具体数值运营配置）。

**设计目的**：体会双方视角、公平性（各当一次收银员/客户）、延长有效练习时间、提升粘性。

## 4. 界面与交互（基础线框）

### 4.1 对话页主结构（竖屏）

```
┌─────────────────────────┐
│ 对方昵称：小张           │
│ 对方信用分：12           │
├─────────────────────────┤
│ 你的任务（收银员）       │
│ ☐ ① 询问用途（点击完成） │
│ ☐ ② 提替代方案（点击）   │
│ ☐ ③ 让客户犹豫（点击）   │
│ ⭐ 最终：阻止取钱         │
├─────────────────────────┤
│ 对方任务进度（可折叠）   │
├─────────────────────────┤
│ [静音] [结束对话]        │
│ 最终决策（仅客户可见）   │
│ [取钱] [不取]            │
└─────────────────────────┘
```

### 4.2 子任务完成流程

1. 用户认为已完成某项子任务 → 点击对应「完成」类按钮。  
2. 本端提示：**「等待对方确认…」**（按钮置灰、防重复提交）。  
3. 对端展示：**「对方声称完成了『xxx』，是否确认？」** → **【确认】/【否认】**。  
4. **确认** → 双方该任务显示为已完成（如划线），能量条增加，可配**短音效**（如「叮」）。  
5. **否认** → 任务保持未完成，记一次争议（MVP 可仅日志，不接仲裁）。

## 5. 防恶意否认与激励

| 手段 | 说明 |
|------|------|
| 信用分 | 恶意否认经申诉核实后**扣分**（与 `产品描述.md` 第 6 节衔接，细则另表）。 |
| 合作分 | 双方**积极确认**对方合理完成时，可额外**合作分/能量**，鼓励配合。 |
| 最终决策 | **不要求**对方确认（见 2.3），避免流程僵局。 |

---

## 6. 任务关联（趣味与策略，可选实现）

| 类型 | 示例 |
|------|------|
| 连锁反应 | A 完成「提替代方案」后，B 的「反驳」任务**解锁**或进入「可确认」态。 |
| 互斥/剧情 | A 完成「让客户犹豫」后，B 点「取钱」前展示文案：「你曾犹豫过，仍坚持取钱吗？」 |
| 情报提示 | A 完成某步后，B 收到轻提示：「收银员正在试图说服你」（不打断语音）。 |

以上以**前端状态机 + 服务端状态**即可实现，**不依赖语音识别**。

## 7. 整局流程小结

| 阶段 | 内容 |
|------|------|
| 准备 | 系统分配角色，展示各自子任务 + 最终目标。 |
| 对话 | 实时语音（声网）；任一方可发起子任务「完成」→ 对方确认/否认/超时。 |
| B 决策 | B 点击「取钱/不取」→ 第一轮结束 → 结算子任务分 + 胜负奖励。 |
| 互换 | 交换角色，重置任务清单，第二轮（限时同第一轮）。 |
| 终局 | 两轮总分比较 → 胜者奖励/勋章。 |

## 8. 不打断对话思路的 UI/UX 原则（推荐）

**目标**：既能操作「自己的任务」、又能处理「对方的确认请求」，又**尽量少打断口语思路**。

### 8.1 原则

- 任务操作**轻量、非强制**：忽略按钮仍可继续说话。  
- 确认请求**非模态**：不用 `alert`、不用全屏居中挡操作；用**顶部横幅**或**底部细条**。  
- **任务清单默认折叠**为小条（如「📋 任务 3/5 ›」），点击再展开**半屏面板**。  
- 中间区域以**通话状态**为主，避免大块文字占屏。

### 8.2 推荐布局（与第 4 节可合并实现）

```
┌─────────────────────────────┐
│ 对方: 小张 ｜ 信用 12        │
│ 主题: 银行取钱 ｜ 倒计时      │
├─────────────────────────────┤
│ （语音通话中，弱提示文案）    │
├─────────────────────────────┤
│ ⚡ 能量 ████░░ 40           │
│ 📋 任务 (3/5) [展开]        │
├─────────────────────────────┤
│ [静音] [结束对话]            │
└─────────────────────────────┘
```

**展开后面板（自下而上或半屏）**

- 我的任务列表 + 各「完成」按钮。  
- **对方任务进度**（可再折叠）：含**待你确认**的条目，便于**事后补确认**（未在 10 秒内点的请求）。  
- 底部：**最终决策**（仅客户角色显示「取钱 / 不取」）。

### 8.3 对方发起「完成」时

- 己方顶部（或底部）滑出**窄条**（约 5 秒自动收起）：  
  `对方称已完成「提替代方案」` → **【确认】【否认】**  
- **超时未点**：视为**未确认**（不加分），请求进入「待确认列表」，不打断当前轮次说话。  
- 可选：**30 秒**后显示「提醒对方」类弱按钮（系统提示音或轻震动，避免弹窗）。

### 8.4 反馈与防误触

- 短音效：己方完成待确认、对方确认成功，各一声区分。  
- 能量条**短动画**反馈奖励。  
- 「完成」后按钮立即**禁用/变灰**，防连点；最终以**对方确认/超时**为准落库。

### 8.5 最终按钮

- 置于展开面板**最下方**、**大触点**、高对比色；点击后进入**结算/结束态**，双方同步结果页。

### 8.6 第二轮

- 仅切换**角色名与任务绑定**，**布局不变**，降低学习成本。

## 9. 后续实现提示（非本文定稿）

- **数据**：需表或 JSON 记录每局 `pair_id`、轮次、角色、子任务 `id`、状态（待确认/已确认/已否认/超时）、时间戳、能量累计；便于复盘与信用申诉。  
- **实时同步**：WebSocket 或由服务端转发 + 客户端短轮询；与声网频道并行。  
- **与现有房间页**：可在 `room.html` 演进为「通话区 + 折叠任务抽屉」，或与 `GET /api/my-pair` 返回的任务结构对齐。

### 9.1 当前前端实现（对照设计稿）

- **页面**：`backend/public/room.html` — 背景 `#1A2233`，顶部三张任务卡（标题 + **+10 ⚡** + 右侧蓝色 **完成**），中部紧凑视频区 + 文案，底部麦克/摄像/挂断。  
- **样式**：`backend/public/css/styles.css`（`.task-confirm-*`、`.partner-confirm-bar`）。  
- **逻辑**：`backend/public/js/room-tasks.js` — 点击「完成」→「等待确认…」→ 底部非模态 **确认/否认** 条（演示用 **10 秒**超时）；接入实时通道后替换为对端真实事件。  
- **扩展内容**：角色与英文提示收于 `<details>` 折叠，减少干扰。
